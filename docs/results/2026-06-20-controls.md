# Tiny Shakespeare Control Comparison

## Protocol

Nine control arms were trained with `configs/ratchet_tiny.toml` (2,000 steps, matched seeds
1337, 1338, 1339) into `runs/controls/`, leaving the six ratchet arms under
`runs/tiny-shakespeare/` untouched. Three controls were run:

- **FP32** — every ratchet matrix replaced with a bias-free `nn.Linear`; all weights trained
  by AdamW. Upper-reference for the architecture.
- **Frozen quinary / frozen septenary** — ratchet codes never move (gradients discarded),
  while the same FP32 embeddings and RMSNorm parameters train. Isolates how much learning
  comes from the support parameters alone versus the ratchet update rule.

**1.8266** (quinary) and **1.7941** (septenary).

## Interpretation

The frozen arms train the *same* FP32 embeddings and RMSNorm parameters as the trainable
ratchet arms, differing only in that the ratchet update never runs. They stall near 3.25
validation loss — well below the ~4.34 random-character reference, but far above the
trainable ratchet's ~1.80. The roughly 1.45-nat gap between frozen and trainable arms is
therefore attributable to the ratchet update rule moving codes, not to the floating-point
support parameters. This is the control the trainability claim needs: the discrete update
is doing the learning.

The trainable ratchet also matched or slightly beat the FP32 reference on this
configuration (≈1.80 vs ≈1.93). At a 2-layer, 128-dim model this is most consistent with a
regularization effect and a support-tuned learning-rate schedule that was not optimized for
the all-FP32 arm; it should be read as *competitive with FP32 here*, not as a general
accuracy advantage, and it says nothing about throughput.

Frozen saturation (quinary 25%, septenary 17%) reflects only the fixed quantized
initialization, since codes never move. It is not comparable to the trainable arms'
end-of-training saturation (quinary ~58%, septenary ~47%), which is driven by the update
rule pinning weights at the code boundaries.
